Проведи полный технический аудит проекта как опытный Senior Software Engineer и архитектор desktop-приложений.

Не ограничивайся поиском очевидных ошибок. Необходимо глубоко проанализировать весь проект и выявить потенциальные проблемы производительности, архитектуры и качества кода.

Представь, что приложение должно активно использоваться несколькими риэлторами каждый день по 8–10 часов без перезапуска. Поэтому необходимо найти всё, что может привести к замедлению работы, росту потребления памяти, зависаниям или усложнению дальнейшей поддержки проекта.

Выполнить полный аудит по следующим направлениям
1. Архитектура проекта

Проверить:

логическую структуру проекта;
разделение ответственности между модулями;
чрезмерно большие файлы;
циклические зависимости;
связанность компонентов;
избыточную вложенность;
нарушения SOLID;
повторяющиеся паттерны.

Для каждой проблемы указать:

где находится;
почему это проблема;
как лучше исправить.
2. Дублирование кода (DRY)

Найти:

одинаковые функции;
похожие функции;
одинаковые проверки;
повторяющиеся обработчики;
одинаковые запросы;
одинаковую логику UI;
одинаковые вычисления;
повторяющиеся стили.

Предложить варианты объединения.

Показать, сколько строк можно убрать после рефакторинга.

3. Производительность

Проанализировать:

медленные алгоритмы;
лишние проходы по массивам;
двойные циклы;
неоптимальные сортировки;
лишние фильтрации;
частые обращения к DOM;
лишние перерисовки;
тяжелые вычисления;
частые JSON.parse();
частые JSON.stringify();
синхронные операции;
блокировку главного потока.

Для каждого места предложить более производительное решение.

4. Утечки памяти (Memory Leak)

Проверить наличие:

не удаляемых EventListener;
забытых setInterval;
забытых setTimeout;
MutationObserver;
ResizeObserver;
IntersectionObserver;
открытых подписок;
незакрытых файлов;
ссылок на DOM после удаления элементов;
глобальных объектов;
накопления данных;
бесконтрольного роста массивов;
накопления истории.

Если есть потенциальная утечка —

показать место и объяснить последствия.

5. Работа с DOM

Проверить:

повторные querySelector;
повторные getElementById;
лишние appendChild;
частые innerHTML;
создание большого количества элементов;
лишние remove/add;
отсутствие DocumentFragment;
отсутствие кэширования элементов.

Предложить оптимизацию.

6. Event Listener

Проверить:

одинаковые обработчики;
повторные подписки;
вложенные обработчики;
обработчики внутри циклов;
анонимные функции;
отсутствие removeEventListener.

Предложить использование Event Delegation там, где это возможно.

7. Работа с памятью

Проверить:

большие объекты;
глубокое копирование;
ненужные копии массивов;
spread оператор;
Object.assign;
Map;
Set;
WeakMap;
WeakSet.

Предложить более эффективные структуры данных.

8. Работа с состоянием приложения

Проверить:

дублирование состояния;
конфликтующие переменные;
глобальные переменные;
гонки данных;
неправильную синхронизацию;
отсутствие централизованного состояния.

Предложить улучшения.

9. UI Performance

Проверить:

плавность интерфейса;
FPS;
количество repaint;
количество reflow;
тяжелые анимации;
box-shadow;
filter;
backdrop-filter;
blur;
layout thrashing.

Если можно ускорить —

предложить изменения.

10. CSS

Проверить:

повторяющиеся правила;
одинаковые цвета;
одинаковые размеры;
одинаковые margin;
одинаковые padding;
одинаковые transition;
лишние селекторы;
слишком длинные селекторы;
unused CSS;
конфликтующие стили.

Предложить рефакторинг.

11. JavaScript

Найти:

неиспользуемые функции;
мертвый код;
неиспользуемые переменные;
лишние импорты;
лишние экспорты;
unreachable code;
console.log;
debugger;
TODO;
FIX ME.
12. Асинхронность

Проверить:

await внутри циклов;
последовательные запросы;
Promise;
Promise.all;
Promise.allSettled;
race conditions;
блокировки.

Предложить оптимизацию.

13. Работа с файлами

Проверить:

открытие Excel;
генерацию документов;
запись файлов;
чтение файлов;
временные файлы.

Убедиться, что ресурсы освобождаются корректно.

14. Безопасность

Проверить:

XSS;
HTML Injection;
eval;
Function();
innerHTML;
внешние данные;
валидацию.
15. Electron (если используется)

Проверить:

preload;
ipcMain;
ipcRenderer;
contextIsolation;
nodeIntegration;
sandbox;
BrowserWindow;
утечки BrowserWindow;
лишние IPC-вызовы.
16. Логи

Проверить:

лишние console.log;
большие объекты в логах;
постоянные сообщения.
17. UX Performance

Проверить:

время запуска;
время открытия сделки;
время импорта Excel;
время генерации договора;
время открытия формы;
время сохранения.

Предложить ускорение.

18. Масштабируемость

Оценить:

Что произойдет если:

будет 500 шаблонов;
20 000 объектов;
10 000 сделок;
100 одновременно открытых документов;
работа без перезапуска весь день.

Какие места начнут тормозить первыми.

19. Код-стайл

Проверить:

читаемость;
именование;
единый стиль;
размер функций;
размер классов;
глубину вложенности;
магические числа;
магические строки.
20. Итоговый отчет

В конце сформировать подробный отчет в формате Markdown.

Разделить найденные проблемы по приоритету:

🔴 Критические

Могут привести к:

зависаниям;
утечкам памяти;
потере данных;
ошибкам генерации документов.
🟠 Высокий приоритет

Заметно ухудшают производительность или усложняют поддержку.

🟡 Средний приоритет

Желательно исправить при ближайшем рефакторинге.

🟢 Низкий приоритет

Косметические улучшения и рекомендации.

Для каждой найденной проблемы обязательно указать:

📁 файл;
📍 точную строку (если возможно);
📖 описание проблемы;
⚠️ возможные последствия;
💡 рекомендуемый способ исправления;
📈 ожидаемый эффект после исправления (ускорение, уменьшение потребления памяти, улучшение читаемости и т. д.).

Важно: не ограничивайся поиском ошибок. Выполни аудит как если бы приложение готовилось к коммерческому релизу с многолетней поддержкой, уделяя особое внимание производительности, стабильности, масштабируемости и удобству дальнейшей разработки.
