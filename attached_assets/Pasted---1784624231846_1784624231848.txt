Необходимо реализовать новую функцию «Выбор риэлтера» в десктопном приложении «Генератор договоров ГермесГарант».

⚠️ Важно:

не менять существующую бизнес-логику приложения;
не ломать существующий интерфейс;
использовать существующий стиль приложения;
новый компонент должен выглядеть как нативная часть интерфейса.
Расположение

Добавить новый элемент в верхний Header.

Разместить его между кнопками «Обновить шаблон» и «О программе».

Новый элемент представляет собой выпадающее меню (Profile Dropdown).

Внешний вид

В закрытом состоянии элемент должен отображать:

круглую фотографию риэлтера (40×40 px);
имя и фамилию;
должность;
стрелку раскрытия вниз.

Например:

◯
Ольга Турко
Директор, риэлтер
               ▼

или

[аватар]   Ольга Турко
           Директор, риэлтер     ▼

Стиль:

современный Fluent Design 3;
Windows 11;
скругление 12 px;
hover;
плавная анимация раскрытия;
использовать существующую цветовую палитру приложения.
Выпадающее меню

При нажатии открывается список сотрудников.

Каждый пункт содержит:

фотографию;
имя;
фамилию;
должность.

Например

○ Ольга Турко
  Директор, риэлтер

○ Иван Иванов
  Агент по операциям с недвижимостью

○ Анна Петрова
  Риэлтер

○ Сергей Ковалев
  Агент по недвижимости

Высота строки около 60 px.

Наведение подсвечивает строку.

Выбранный сотрудник отмечается галочкой справа.

Источник данных

Все фотографии брать из

assets/

Не использовать абсолютные пути.

Все сотрудники должны храниться в отдельном файле.

Например

data/realtors.json

или аналогичной существующей структуре проекта.

Структура объекта:

[
  {
    "id": "turko",
    "firstName": "Ольга",
    "lastName": "Турко",
    "position": "Директор, риэлтер",
    "photo": "assets/realtors/turko.jpg"
  }
]

Использовать именно этот источник данных.

Не хардкодить сотрудников в коде.

Логика выбора

После выбора сотрудника необходимо:

закрыть меню;
обновить Header;
показать выбранного сотрудника;
сохранить выбор.

При следующем запуске приложения должен автоматически отображаться последний выбранный риэлтер.

Сохранение

Выбор должен сохраняться между запусками приложения.

Использовать существующий механизм хранения настроек проекта.

Если такого механизма нет — реализовать его.

Допустимые варианты:

electron-store
localStorage (если используется)
файл settings.json
существующий SettingsService

Не создавать новый механизм, если уже существует аналогичный.

Первый запуск

Если пользователь еще ничего не выбирал:

автоматически выбрать первого сотрудника из списка.

Никаких пустых состояний быть не должно.

Связь с генерацией документов

Выбранный риэлтер становится активным риэлтером приложения.

В дальнейшем именно его данные должны использоваться всеми существующими механизмами формирования договоров.

То есть необходимо реализовать единый источник правды (Single Source of Truth).

Любой модуль приложения, который запрашивает текущего риэлтера, должен получать выбранного пользователя.

Не дублировать состояние.

Архитектура

Создать отдельный сервис.

Например

RealtorService

который отвечает за:

загрузку списка сотрудников;
получение текущего сотрудника;
изменение текущего сотрудника;
сохранение выбора;
уведомление интерфейса об изменении.

Не смешивать UI и бизнес-логику.

Требования к коду

Использовать существующий стиль проекта.

Не менять архитектуру.

Не создавать лишних зависимостей.

Максимально переиспользовать существующие компоненты.

Если уже есть:

Dropdown;
Menu;
Avatar;
SettingsService;
Store;

использовать именно их.

UX

Добавить:

✔ плавное открытие меню

✔ плавное закрытие

✔ hover

✔ focus

✔ навигацию клавиатурой

↑ ↓

Enter

Esc

Tab

Accessibility

Добавить:

aria-label
role
keyboard navigation
правильный focus ring
Производительность

Фотографии должны загружаться лениво.

После первой загрузки использовать кэш.

Не выполнять повторную загрузку изображения без необходимости.

Обработка ошибок

Если фотография отсутствует:

показать аватар-заглушку с инициалами.

Например

ОТ

Если JSON отсутствует:

вывести сообщение в лог.

Приложение не должно завершаться с ошибкой.

Что необходимо проверить после реализации

Проверить все пользовательские сценарии:

отображение списка;
выбор сотрудника мышью;
выбор клавиатурой;
сохранение между запусками;
восстановление после перезапуска;
отсутствие мерцания интерфейса;
корректное отображение фотографий;
работу при отсутствии фотографии;
работу при отсутствии JSON;
работу после обновления шаблонов;
работу после открытия нового Excel-файла;
работу после генерации документов.

Убедиться, что новая функциональность не нарушает существующую бизнес-логику приложения и полностью интегрирована в текущую архитектуру проекта. В конце работы провести рефакторинг кода новой функциональности, удалить дублирование, использовать единый источник состояния выбранного риэлтера и привести реализацию к промышленному качеству.
